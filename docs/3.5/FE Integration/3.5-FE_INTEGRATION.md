### Mapping APIs

#### Education-License Mappings
- **POST /mst-education-license-mappings** - Create education-license mapping
  - Accessible by: SYS_ADMIN (all schools) or PIC_SCHOOL (assigned schools only)
  - Purpose: Map education criteria (school + major + degree + diploma level) to license
  
- **GET /mst-education-license-mappings/search** - Search mappings with filters, pagination, scope filtering
  - Accessible by: SYS_ADMIN (all schools) or PIC_SCHOOL (assigned schools only)
  
- **GET /mst-education-license-mappings/:id** - Get mapping by ID
  - Accessible by: SYS_ADMIN or PIC_SCHOOL (if assigned to mapping's school)
  
- **PUT /mst-education-license-mappings/:id** - Update mapping
  - Accessible by: SYS_ADMIN or PIC_SCHOOL (if assigned to mapping's school)
  - Note: Triggers retroactive processing if mapping criteria changed
  
- **DELETE /mst-education-license-mappings/:id** - Soft delete mapping

- **GET /mst-education-license-mappings/template/download** - Download Excel template for bulk upload

- **POST /mst-education-license-mappings/upload** - Upload Excel file for bulk mapping

### User Data APIs

#### User Educations
- **POST /user-educations** - Create new user education
  - Accessible by: Authenticated users
  - Purpose: Create education record (defaults to `approval_state = WAITING_APPROVAL`)
  - **Auto-Approval Flow:** 
    - If matching `pending_student_verification` exists (same `student_id + school_id`)
    - System automatically approves: `approval_state = APPROVED`, `approval_by = 'system'`, `is_verified = true`
    - Pending entry is deleted
    - Triggers `EDUCATION_VERIFICATION_QUEUE` for license granting
  - **Pending Certificate Creation:**
    - System finds all matching education-license mappings
    - Creates pending certificates with `approval_state = WAITING_APPROVAL`
    - Links certificates to education via `user_education_id`
  
- **GET /user-educations/search** - Search educations with filters, pagination
  - Accessible by: Authenticated users
  
- **PATCH /user-educations/:id** - Update education
  - Accessible by: Authenticated users
  - **Restriction:** Can only update if `approval_state = WAITING_APPROVAL`
  - Cannot update if `approval_state = APPROVED` or `REJECT` → BadRequestException
  
- **DELETE /user-educations/:id** - Soft delete education
  - Accessible by: Authenticated users
  - Also deletes linked certificates via cascade
  
- **PATCH /user-educations/:id/approval** - Approve/reject education (Admin only)
  - Accessible by: Admin users
  - Purpose: Set `approval_state` to `APPROVED` or `REJECT`
  - **If APPROVED:** Triggers `EDUCATION_VERIFICATION_QUEUE` for automatic license granting
  - **Validation:** Cannot change if already in final state (APPROVED/REJECT)
  
- **GET /user-educations/need-approval** - Get educations needing approval (Admin only)
  - Accessible by: Admin users
  - Purpose: List all educations with approval_state = WAITING_APPROVAL
  
- **GET /user-educations/students-by-school/:schoolId** - Get students by school with verification status
  - Accessible by: PIC_SCHOOL, SYS_ADMIN, Admin users
  - Purpose: Returns aggregated list of students for a given school, categorized by verification status
  - Response includes enriched profile fields (full_name, photo_url, email, phone_num, major)
  - Status values: VERIFIED (approval_state=APPROVED), NEED_VERIFICATION (approval_state=WAITING_APPROVAL), NOT_VERIFIED (approval_state=REJECT), USER_NOT_FOUND (exists in pending_student_verifications)
  - Profile fields are populated from user data (for matched users) or from pending entries (for USER_NOT_FOUND)
  - Supports pagination (page, limit)

#### User Certificates
- **POST /user-certificates** - Create new user certificate
  - Accessible by: Authenticated users
  - Purpose: Create certificate record (defaults to `approval_state = WAITING_APPROVAL`)
  - **Important:** 
    - `user_id` is **optional** - if not provided, automatically derived from authentication token (`req.user.id`)
    - `mst_license_id` is **required** (must create/get master license first)
    - License details (`license_number`, `license_name`, `issuing_organization`, `issue_date`) are **automatically populated** from `mst_license` entity - do NOT include these fields in request body
    - Frontend flow: Create/get `mst_license` first via `POST /mst-licenses` or `GET /mst-licenses/:id`, then use the license ID when creating user certificate
    - Request body fields: `mst_license_id` (required), `user_id` (optional), `no_expiry`, `expiry_date`, `description`, `file_url`, `certificate_level`
  - **Retroactive Check:**
    - Triggers `RETROACTIVE_EDUCATION_LICENSE_QUEUE` to check if user's approved educations should auto-approve this certificate
    - If mapping exists, certificate auto-approved: `approval_state = APPROVED`, `approval_by = 'system'`
  
- **GET /user-certificates/search** - Search certificates with filters, pagination
  - Accessible by: Authenticated users
  
- **GET /user-certificates/:id** - Get certificate by ID
  - Accessible by: Authenticated users
  
- **PATCH /user-certificates/:id** - Update certificate
  - Accessible by: Authenticated users
  - **Restriction:** Can only update if `approval_state = WAITING_APPROVAL`
  - Cannot update if `approval_state = APPROVED` or `REJECT` → BadRequestException
  - The following fields are **READ-ONLY** (ignored if provided):
    - `approval_state`, `approval_by` - Only set via approval flow
    - `license_number`, `license_name`, `issuing_organization`, `issue_date`, `mst_license_id` - Populated from `mst_license` entity
    - `is_verified` - Set by system grants
  
- **DELETE /user-certificates/:id** - Soft delete certificate
  - Accessible by: Authenticated users
  - Cannot delete if linked to active user education (must delete education first)
  
- **PATCH /user-certificates/:id/approval** - Approve/reject certificate (Admin only)
  - Accessible by: Admin users
  - Purpose: Set `approval_state` to `APPROVED` or `REJECT`
  - **If APPROVED:** Triggers `CERTIFICATE_VERIFICATION_QUEUE` for automatic skill granting (if `mst_license_id` exists)
  - **Validation:** Cannot change if already in final state (APPROVED/REJECT)
  
- **GET /user-certificates/need-approval** - Get certificates needing approval (Admin only)
  - Accessible by: Admin users
  - Purpose: List all certificates with approval_state = WAITING_APPROVAL
  
- **POST /user-certificates/migrate-to-mst-licenses** - Migrate existing certificates to mst_licenses (Internal/Admin only)
  - Accessible by: Admin users
  - Purpose: One-time migration to create mst_license entries from existing user_certificates

#### Pending Student Verifications
- **POST /pending-student-verifications** - Create pending student verification (Admin only)
  - Accessible by: SYS_ADMIN, Admin, PIC_SCHOOL
  - Purpose: Create buffer entry for students uploaded by admin but not yet registered in system
  - Fields: student_id (required), school_id (required), full_name, photo_url, email, phone_num, major (all optional)
  - **Validation:** 
    - Cannot create if user_education already exists with same student_id + school_id
    - Error messages: "This student already approved" (if approval_state=APPROVED), "Data already registered & need approval" (if approval_state=WAITING_APPROVAL)
    - Cannot create duplicate pending entry (same student_id + school_id)
  
- **GET /pending-student-verifications/search** - Search pending student verifications
  - Accessible by: SYS_ADMIN, Admin, PIC_SCHOOL
  - Supports pagination, filtering, sorting, Excel export
  
- **PATCH /pending-student-verifications/:id** - Update pending student verification
  - Accessible by: SYS_ADMIN, Admin, PIC_SCHOOL
  - Same validation rules as create
  
- **DELETE /pending-student-verifications/:id** - Soft delete pending student verification
  - Accessible by: SYS_ADMIN, Admin, PIC_SCHOOL
  
- **GET /pending-student-verifications/template/download** - Download Excel template
  - Accessible by: SYS_ADMIN, Admin, PIC_SCHOOL
  - Template columns: student_id, school_name, full_name, photo_url, email, phone_num, major
  
- **POST /pending-student-verifications/upload** - Upload Excel file for bulk import
  - Accessible by: SYS_ADMIN, Admin, PIC_SCHOOL
  - Accepts multipart/form-data with Excel file
  - Returns success_count, error_count, errors[] with row numbers and messages
  - Fails fast if school_name not found in mst_schools

### Automation Flow Triggers

These endpoints trigger background queue processing automatically:

#### User Data Triggers (Forward Flow: User → Auto-Approval)

- **POST /user-educations** - Always triggers:
  1. Pending certificate creation for all matching mappings
  2. If matching `pending_student_verification` exists → auto-approves education → triggers education verification queue
  
- **PATCH /user-educations/:id/approval** - If `approval_state` set to `APPROVED`, triggers education verification queue (grants licenses automatically)
  
- **POST /user-certificates** - Triggers retroactive check to see if user's approved educations should auto-approve this certificate
  
- **PATCH /user-certificates/:id/approval** - If `approval_state` set to `APPROVED` AND `mst_license_id` exists, triggers certificate verification queue (grants skills automatically)
  
- **POST /user-skills** - Triggers retroactive check to see if user's approved certificates should auto-approve this skill
  
- **POST /users/register-candidate** - Triggers informal certificate user processing queue (matches user with unprocessed mappings)
  
- **PUT /users/:id** - If email/phone changed, triggers informal certificate user processing queue (matches user with unprocessed mappings)

#### Mapping Triggers (Reverse Flow: Mapping → Find Users)

- **POST /mst-education-license-mappings** - Triggers retroactive education-license queue (grants licenses to existing approved educations)
  
- **PUT /mst-education-license-mappings/:id** - If mapping criteria changed, triggers retroactive education-license queue
  
- **POST /mst-license-skill-mappings** - Triggers retroactive license-skill queue (grants skills to existing approved certificates)
  
- **PUT /mst-license-skill-mappings/:id** - If `license_id` or `skill_id` changed, triggers retroactive license-skill queue
  
- **POST /mst-informal-certificate-mappings** - Triggers informal certificate processing queue (finds matching users and grants certificates)
  
- **PUT /mst-informal-certificate-mappings/:id** - If status still `UNPROCESSED`, re-triggers processing queue

### Notes

- All endpoints require Bearer token authentication (`@ApiBearerAuth()`)
- Approval endpoints (`/approval`) are restricted to Admin users only
- PIC_SCHOOL role can only manage mappings for their assigned schools
- SYS_ADMIN can manage all mappings regardless of school
- All search endpoints support pagination, filtering, sorting, and Excel export
- Retroactive processing runs asynchronously via Bull queues

#### Approval State System
- **State Flow:** `WAITING_APPROVAL` → `APPROVED` or `REJECT` (final states cannot be changed)
- **Automation Trigger:** `approval_state = APPROVED` triggers background automation (license/skill granting)
- **FE Display:** Use `is_verified` field to show verified badges/states
  - `is_verified = true` means `approval_state = APPROVED` (mirrors the state)
  - Backend always sets both fields together for consistency
- **Edit Restrictions:** 
  - Can only edit user data (educations, certificates, skills) if `approval_state = WAITING_APPROVAL`
  - Cannot edit if `approval_state = APPROVED` or `REJECT` → API returns BadRequestException

#### Pending Certificate Creation
- When user creates education, system automatically creates **pending certificates** linked to that education
- Pending certificates have `approval_state = WAITING_APPROVAL` and `user_education_id` set
- When education is approved, these certificates are automatically approved too
- This ensures referential integrity between education and granted licenses

#### Retroactive Processing (Bi-Directional)
- **Direction A:** When admin creates mapping → system finds existing approved user data and grants licenses/skills
- **Direction B:** When user creates certificate/skill → system checks if existing education/certificate should auto-approve it
- Both directions run asynchronously via Bull queues

#### Pending Student Verifications
- Buffer table for students uploaded by admin before user registration
- When user creates education with matching `student_id + school_id`, education is auto-approved and pending entry is deleted
- **Students by School Endpoint:** Aggregates data from both `user_educations` and `pending_student_verifications` tables
- Priority order: VERIFIED > NEED_VERIFICATION > NOT_VERIFIED > USER_NOT_FOUND
